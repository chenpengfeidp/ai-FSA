## 1. Provider capability analysis (API-Sports Football)

Provider under study: **API-Football** via existing F.1 host `https://v3.football.api-sports.io` (`football:api-sports`).

Plan facts (unchanged from research): Free and Paid expose the **same endpoint families**; differentiation is **daily quota** and **historical season depth**, not feature paywalls. League truth is always `/leagues` season **`coverage`** flags.

| # | Question | Answer |

| 2 | 免费版是否支持 | **Yes** (quota-limited: 100 req/day) |
| 3 | 付费版是否支持 | **Yes** |
| 4 | Endpoint | `GET /injuries` (`fixture`, `team`, `league`+`season`, `player`, `date`) |

## 3. Sprint Scope (MoSCoW)

### Must

1. Fixture-scoped **Injury** facts from API-Football `/injuries` when coverage/data allows.
2. Fixture-scoped **Suspension** facts from the same feed (classified distinctly in payload / UI).
3. Composed **Availability** Workspace + Report surface citing those Facts.
4. **Honest absence**: coverage false, empty response, or provider failure must **not** become “no injuries / everyone available” (align doc 41).
5. Recorded cassette samples: happy-path (injury + suspension rows) + absence-path.
6. Registry: `injury.ingestImplemented=true` only when the path is wired.
7. Preserve analyze minimum: MATCH_INFO + TEAM_FORM×2 + STATISTICS×2 still sufficient; Availability **never** required for Projection.
8. Sprint artifacts cite roadmap **F1.1** / sprint **F1.1D**.
9. Feature / Rule / Projection packages remain **untouched**.

### Should

1. Distinct UI grouping: Injury vs Suspension within Availability.
2. Provenance / provider source lines consistent with F1.1A / Venue / Player.
3. Optional light join to existing PLAYER Evidence (name/id) when ids match — presentation only.
4. Explicit limitation copy in Report when injuries coverage unavailable for the league/season.

### Could

1. Include `/sidelined` historical narrative (defer if it expands scope).
2. Show count summaries (e.g. N injured / M suspended) derived only from Evidence rows.
3. Cap live injury fan-out for Free-tier demos (recorded-first validation).

### Won't

1. **Expected / probable Lineup** (no provider Fact source — do not invent; do not misuse `/predictions`).
2. Confirmed Lineup ingest (→ **F1.1E**).
3. Referee / Coach / richer Form / Weather.
4. Player Statistics / Ratings.
5. Feature / Rule / Projection math changes.
6. Making Availability required for `analyzable`.
7. New Architecture documents or Engines.
8. Redis / BullMQ polling platform.
9. Full Squad Availability matrix product.

## 4. Sprint Backlog (Vertical Slices)

All slices follow:

```text
Provider → Normalizer → Evidence → Workspace → Report
```

**Excluded from every slice:** Feature, Rule, Projection.

Technical note: Injury and Suspension share `GET /injuries`. Coding may use **one provider fetch** and split presentation/Evidence rows; backlog slices remain reviewable product increments.

### Slice 1 — Injury Evidence

```text
Provider (/injuries → injury rows)
  → Normalizer
  → Evidence (INJURY, type=injury / reason)
  → Workspace
  → Report
```

| Item | Acceptance sketch |
|---|---|
| Map fixture injuries with player id/name, team, reason, observed context | Present when provider returns injury rows |
| Cassette happy-path | CI without live key |
| Timeline | Type `INJURY` visible with provider source |
| Absence | Empty/coverage → no invented “zero injuries” claim |

### Slice 2 — Suspension Evidence

```text
Provider (/injuries → suspension rows)
  → Normalizer
  → Evidence (INJURY family with suspension classification — or agreed payload discriminant)
  → Workspace
  → Report
```

| Item | Acceptance sketch |
|---|---|
| Suspension rows distinct from injury in payload + UI labels | Analyst can tell Injury vs Suspension |
| Same honest-absence rules as Slice 1 | No silent empty success |
| Cassette includes at least one suspension row | CI coverage |

### Slice 3 — Availability Evidence (composition + honesty)

```text
Provider (composed Availability view from Slice 1+2 outputs)
  → Normalizer / Evidence selection (no new math)
  → Evidence set citation
  → Workspace Availability section
  → Report Availability section + limitations
```

| Item | Acceptance sketch |
|---|---|
| Single Availability section listing injured + suspended | Primary F1.1D product surface |
| Missing-state honesty (coverage / empty / failure) | Explicit limitation text |
| Optional join to PLAYER basic ids | Display only |
| Analyze still succeeds without Availability | Backward compatible |

### Delivery order

1. Slice 1 (Injury)  
2. Slice 2 (Suspension) — may land with Slice 1 if shared feed keeps risk low  
3. Slice 3 (Availability composition + absence UX)  
4. Implementation Report + Demo notes (same pattern as F1.1C-1)

## 5. Risk Assessment

### Provider Coverage Risk

| Risk | Impact | Mitigation |
|---|---|---|
| `coverage.injuries=false` for target leagues | Empty Availability often | Check coverage; show explicit “coverage unavailable”; do not invent |
| Sparse fixture injury lists even when coverage true | Thin demos | Prefer leagues with known coverage; rely on recorded cassettes for CI |

### League Coverage Risk

| Risk | Impact | Mitigation |
|---|---|---|
| FAS default league set uneven for injuries | Live demos inconsistent | Document supported demo leagues; recorded K League (or chosen) cassette as truth for CI |
| Season rollover coverage changes | Sudden empties | Treat as absence honesty, not regression of Projection |

### Paid Tier Risk

| Risk | Impact | Mitigation |
|---|---|---|
| Free **100 req/day** — enrich fan-out already high; `/injuries` adds calls | Live demos burn quota | Recorded-first CI; cap live; paid tier for interactive multi-match demos |
| Historical `/sidelined` depth | Temptation to expand scope | Keep `/sidelined` in Could / Won't for MVP |

### Provider Lock Risk

| Risk | Impact | Mitigation |
|---|---|---|
| Mapping too close to API-Football field names | Harder replaceability | Normalize into FAS Availability / INJURY payload semantics; retain aliases in provenance only (doc 41) |
| Treating `/predictions` as lineup/availability | Epistemic breach | Explicit Won't; Category firewall |

### Backward Compatibility

| Risk | Impact | Mitigation |
|---|---|---|
| Requiring INJURY for analyze | Breaks F.1 path | Availability optional; minimum evidence unchanged |
| Payload shape changes breaking Feature | Unintended math coupling | Do not teach Feature/Rule to read Availability in F1.1D |
| Old Evidence rows without injury | Fine | Additive Evidence only |

## 6. Sprint Exit Criteria

Sprint **F1.1D** is complete when **all** of the following hold:

1. **Provider path:** Live and/or recorded Football path can supply injury + suspension rows for a demo fixture without inventing data.
2. **Normalizer → Evidence:** `INJURY` Evidence (including suspension discriminant) is produced with provider metadata (`providerId`, provenance, freshness/confidence policy consistent with F1.1A).
3. **Workspace:** Availability section shows Injury and Suspension when Evidence exists.
4. **Report:** Explainable report cites Availability Facts; Evidence timeline shows `INJURY` (and suspension labeling).
5. **Honest absence:** At least one validated path where missing coverage/empty feed yields **explicit limitation**, not silent “no absences.”
6. **Registry:** `football:api-sports` capability `injury.ingestImplemented === true`.
7. **Regression:** `@fas/feature`, `@fas/rule`, `@fas/analysis` tests pass **without** code changes in those packages.
8. **Analyze compatibility:** Projection still runs on F.1 minimum evidence alone.
9. **Artifacts:** Implementation Report (and demo notes if used) cite roadmap F1.1 / F1.1D; Expected Lineup remains unimplemented.
10. **Scope integrity:** No Feature / Rule / Projection / Expected Lineup / confirmed Lineup / Referee work landed as “drive-by.”

Non-goals that do **not** block F1.1D exit: confirmed Lineup, Referee, richer Form, Squad Availability matrix, math wiring of absences.

## 7. Recommendation

### Primary recommendation

Proceed to Coding for **F1.1D Availability (Injury + Suspension)** as three backlog slices above, reusing F1.1B-1 / F1.1C-1 vertical-slice discipline.

### Expected Lineup

**Not suitable for MVP. Place in Won't.**

Reasons:

1. API-Football has **no** Expected / probable XI endpoint.
2. `/predictions` is a **Prediction-category** signal, not Lineup Fact (doc 41 Category firewall).
3. Inventing Expected Lineup would violate Evidence-first / “AI drafts; FAS validates; human publishes” epistemic rules.
4. Confirmed Lineup (when published) belongs in a later slice (**F1.1E**), with explicit missing/not-published states — still not “expected.”

### Suggested Coding kickoff order

1. Thin Spec notes only if Coding needs field-level contracts (optional; not a new Architecture doc).  
2. Implement Slice 1 → 2 → 3.  
3. Tests + recorded cassettes + Implementation Report.  
4. Re-open F1.1 Review matrix after delivery.
