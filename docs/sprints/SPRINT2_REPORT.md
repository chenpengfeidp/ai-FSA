## Decisions Made

- The applications were assembled manually instead of retaining generator examples or duplicate tooling.
- Direct dependencies are exact-pinned in each application manifest.
- The API uses one minimal NestJS module and one operational controller. It exposes only:
  - `GET /`
  - `GET /health/live`
  - `GET /health/ready`
  - `GET /version`
- The API binds to `127.0.0.1:3001` by default to avoid unintended network exposure and to coexist with the web application.
- Readiness reports the application-shell state only. Database readiness is intentionally absent because Sprint 2 prohibits database and Prisma integration.
- The web application uses the Next.js App Router with one homepage and no styling framework, authentication, or additional routes.
- Next.js generated types are reproducible through `next typegen` and are not committed.
- The worker uses a standalone NestJS application context, logs `Worker started.`, and closes cleanly. No artificial idle loop, queue, scheduler, Redis client, or background job was introduced.
- No shared package was created because the three shells do not yet have a justified shared application contract.
- TypeScript `7.0.2` failed the required compatibility gate because Nest CLI 11 requires the programmatic compiler API that TypeScript 7.0 does not expose. The approved fallback was applied using exact-pinned TypeScript `6.0.3`.
- The only permitted dependency build script is `sharp`, a transitive dependency used by Next.js.

## Validation Performed

The following checks succeeded with Node.js `24.18.0` and pnpm `11.13.0`:

- `pnpm install`
- `pnpm install --frozen-lockfile`
- `pnpm workspace:check`
  - Detected `@fas/api`, `@fas/web`, and `@fas/worker`.
- `pnpm typecheck`
  - Passed for all three applications.
  - Next.js route types were generated successfully.
- `pnpm build`
  - NestJS API build passed.
  - NestJS worker build passed.
  - Next.js production build passed and produced only the root static page plus framework fallback behavior.
- `pnpm dev:api`
  - Started successfully on `http://127.0.0.1:3001`.
  - All four allowed endpoints returned successful JSON responses.
- `pnpm dev:web`
  - Started successfully on `http://127.0.0.1:3000`.
  - The homepage returned both required messages.
- `pnpm dev:worker`
  - Compiled without errors and logged `Worker started.`
- `pnpm --filter @fas/worker start`
  - Production worker bootstrap logged the required message and exited cleanly.
- IDE diagnostics reported no errors in the three application directories.

Development processes used for runtime validation were stopped after the checks completed.
