## Implementation Report

### Goal

End-to-end **Player Basic Information** evidence:

`Provider → Normalizer → Evidence → Workspace → Report`

without changing Rule / Projection / Feature math.

### Delivered path

| Stage | Change |
|---|---|
| Provider | `FootballPlayer` on `FootballMatchBundle`; map `/players/squads` (live, capped); recorded cassettes include basic players; `toEvidenceMatchShape` emits optional `players[]` |
| Normalizer | Optional `parsePlayers` → Evidence type **`PLAYER`** with provider metadata |
| Evidence | New type `PLAYER`; registry capability `player.ingestImplemented=true` for `football:api-sports` |
| Workspace | Players section (home/away) + Evidence timeline PLAYER icon/title |
| Report | Explainable report surfaces players from Evidence (presentation only; not Projection) |

### Fields (basic identity only)

- Player Id
- Name
- Team (id, name, side)
- Position (when provider has it)
- Number (when provider has it)
- Nationality (when provider has it)
- Photo (when provider has it)

### Explicitly not changed / not implemented

- Player Statistics, Rating, Injury, Suspension, Expected Lineup
- `@fas/feature` Feature extraction
- `@fas/rule` Rule evaluation
- `@fas/analysis` DeterministicMatchProjection

### Honest absence

If `players` is missing or empty from provider shape, **no** `PLAYER` Evidence is emitted (squad is not invented).

|---|---|

| Postgres envelope | `PLAYER` stored like other types in `value_json`; no schema migration |

---

## Migration Notes

- No Prisma / SQL migration.
- Additive Evidence type `PLAYER` in domain + API DTO enum + web `EvidenceType`.
- Live catalog may call `/players/squads` for home and away (max 12 each); empty squad responses do not fail the bundle.
- Consumers that exhaustively switch on `EvidenceType` must accept `PLAYER` (web titles / timeline updated).
