## 4. Next phase recommendation — F1.1D

### Recommendation

**Sprint F1.1D — Injury / Suspension (Availability) Evidence vertical slice**

Follow the same pattern as F1.1B-1 / F1.1C-1:

```text
Provider → Normalizer → Evidence → Workspace → Report
```

Keep **Feature / Rule / Projection** untouched until a later gated math sprint.

### Why this next

1. Roadmap F1.1 **Must**: availability is first-class analytical input.
2. Builds on **Player basic** identity already in Evidence (F1.1C-1).
3. Provider feed exists (`/injuries`; suspension reasons in same family) — not blocked like Expected Lineup.
4. Highest acceptance gap after Venue + Player: analysts still cannot see who is missing.

### Suggested F1.1D scope (product, not design)

| In | Out |
|---|---|
| Injury + suspension rows as Facts (honest absence when coverage/empty) | Player Statistics / Rating |
| Evidence population of `INJURY` (or agreed availability payload) | Expected Lineup |
| Workspace + Report availability section | Feature / Rule / Projection changes |
| Registry `injury.ingestImplemented=true` when wired | Coach / Referee (separate thin slices) |
| Recorded cassette samples for CI | New Architecture / Engines |

### Suggested sequence after F1.1D

| Id | Theme | Rationale |
|---|---|---|
| **F1.1E** | Confirmed Lineup + Missing / not-published states | Completes “who plays”; pairs with availability |
| **F1.1F** | Referee identity (match context) | Small identity slice; tendencies later / out of scope |
| **F1.1G** | Richer Recent Form | Completes roadmap “richer form”; still Evidence-first |
| Later / other | Coach name; Player stats; Weather; math wiring of availability into Feature→Rule | Not F1.1D |

### Stop conditions for F1.1 overall

Treat roadmap Sprint **F1.1** as complete only when acceptance in doc 40 is met for supported matches: lineup **or** explicit missing, injury/suspension **or** explicit missing, referee when published, recent form (including richer form if still in scope), without inventing Expected Lineup.
